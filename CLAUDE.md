# WPR Adoptable Pets Widget

An automated data pipeline and frontend widget that scrapes adoptable pet listings from five central Wisconsin shelters and presents them in a single, branded Wausau Pilot & Review interface. The widget runs as a static HTML page deployed to GitHub Pages and embedded in WordPress via iframe.

**Repository:** `RowanFlynnPilot/wpr-adopt-widget`
**Live URL:** `https://rowanflynnpilot.github.io/wpr-adopt-widget/adopt-widget.html`
**Data endpoint:** `https://rowanflynnpilot.github.io/wpr-adopt-widget/pet-data.json`

## Shelters & Scraper Details

### 1. Marathon County Humane Society (`marathon`)
- **Source:** Adoptapet.com shelter page
- **Shelter ID:** `77626-humane-society-of-marathon-county-wausau-wisconsin`
- **Method:** `scrapeAdoptapet()` — Puppeteer with client-side pagination (clicks Next/numbered buttons; URL `?page=N` doesn't work)
- **Fallback:** If card scraping finds ≤3 pets, a raw HTML fallback extracts pet URLs from the page source (RSC payloads, script data)

### 2. Clark County Humane Society (`clark`)
- **Source:** Petfinder.com member page
- **Slug:** `neillsville/clark-county-humane-society-wi34`
- **Method:** `scrapePetfinder()` — Scrapes `a[href*="/details/"]` cards, parses alt text for name/breed/age/gender

### 3. Adams County Humane Society (`adams`)
- **Source:** Adoptapet.com shelter page
- **Shelter ID:** `76343-adams-county-humane-society-friendship-wisconsin`
- **Method:** `scrapeAdoptapet()` (same as Marathon)

### 4. Lincoln County Humane Society (`lincoln`)
- **Source:** Adoptapet.com shelter page (migrated from furrypets.com which was unreliable)
- **Shelter ID:** `66070-lincoln-county-humane-society-merrill-wisconsin`
- **Method:** `scrapeAdoptapet()` (same as Marathon/Adams)
- **Note:** `scrapeLincoln()` function still exists in the codebase for the old furrypets.com approach but is not called from `main()`

### 5. New Life Pet Adoption Center (`nlpac`)
- **Source:** nlpac.com/pets (direct HTML)
- **Method:** `scrapeNlpac()` — Two-pass: first scrapes listing page for pet links (`a[href*="/q/pets/"]`), then visits each detail page to extract name, species, breed, age, bio, and photo
- **Species detection:** Parses "Animal Type" field from structured `<li>` data on detail pages

### Cross-Shelter Deduplication
After all scrapers run, `main()` deduplicates by URL across shelters. Priority order: marathon → clark → adams → lincoln → nlpac (first occurrence wins). This handles Adoptapet cross-listing the same pet under multiple nearby shelters.

## Scraper Internals

### Anti-Bot Measures
All scrapers use a shared `makePage(browser)` helper that applies:
- Realistic Chrome User-Agent string
- `navigator.webdriver = false`
- Fake plugins array and language list
- `window.chrome.runtime` stub
- 1280×900 viewport

### Adoptapet Scraper (`scrapeAdoptapet`)
This is the most complex scraper (~270 lines). Key behaviors:
- **Pagination:** Client-side only. Detects "X - Y of Z" text to determine total count, finds and clicks numbered page buttons or "Next" links, waits for range indicator to update before scraping next page.
- **Card parsing:** Finds `a[href*="/pet/"]` links. Extracts name from `img[alt^="Photo of"]` or first text line (with cleanup for concatenated blobs like "AdaDomestic ShorthairFemale, 11 mos"). Breed extracted from second text line or parsed from fullText.
- **Bio fetching:** After card scraping, visits each pet's detail page to extract bio text (from `<p>` elements in main/article/content containers) and breed (from `<dt>Breed</dt><dd>` pairs, page title, or og:description meta).
- **Fallback:** If only ≤3 pets found from cards, re-fetches the page and regex-extracts `/pet/(\d+)-([a-z-]+)` URLs from raw HTML (catches server-rendered data that JS hydration may have removed).
- **Image optimization:** Rewrites Adoptapet image URLs to use their Cloudinary CDN with `w_400,ar_4:3,dpr_2` parameters.

### Data Normalization
All scrapers output a standard pet object:
```json
{
  "name": "Sonny",
  "species": "Dog",       // "Dog", "Cat", or "Other"
  "breed": "Schnauzer (Miniature)",
  "age": "3 yrs",
  "gender": "Male",
  "bio": "Short description...",
  "photo": "https://...",  // null if unavailable
  "url": "https://..."     // link to pet's detail page
}
```

Species classification uses keyword matching: cat breeds (shorthair, longhair, siamese, etc.) and URL patterns (`-cat$`) → Cat; small animals (rabbit, guinea pig, hamster, etc.) → Other; everything else → Dog.

## Frontend Widget

### Tech Stack
- **Single-file HTML** — all CSS and JS inline, no build step
- **Fonts:** Playfair Display (headings), Source Sans 3 (body) via Google Fonts
- **No frameworks** — vanilla JS with DOM manipulation

### WPR Brand Tokens
```css
--teal: #4a9e8e;
--teal-dark: #3a7d70;
--teal-light: #5fb8a6;
--teal-bg: #e8f4f1;
--ink: #1a1a1a;
--warm-white: #fafaf8;
--sand: #f2efea;
```

### Data Loading Strategy
1. Widget starts with **hardcoded fallback data** (`FALLBACK_DATA` const) so it always renders immediately
2. `loadLiveData()` fetches `pet-data.json` from GitHub Pages in the background (cache-busted with `?t=Date.now()`)
3. Merges live data per-shelter: if a shelter has live pets, replaces fallback; if empty, keeps fallback
4. Re-renders after merge

### Filters
- **Shelter filter:** All Shelters, Marathon County, Clark County, Adams County, Lincoln County, New Life PAC
- **Animal type filter:** All Animals, Dogs, Cats, Other (Other button auto-hides when no small animals exist)
- State tracked in `activeShelter` and `activeAnimal` globals; `render()` rebuilds all DOM

### Pet Cards
- Photo with smart cropping (`smartPosition()` — canvas-based edge detection to find focal third of image for `object-position`)
- Fallback emoji placeholder when no photo available
- Name, breed + age, bio excerpt, species/gender tags
- Click anywhere on card → opens pet detail page in new tab
- Share button with email, SMS, and copy-link options

### Widget Sections (per shelter)
- Branded header with shelter icon, name, location, and "Visit Website →" link
- Pet grid (responsive CSS grid)
- "View all at [shelter]" link at bottom
- Status banner showing total count + active filters

## GitHub Actions Pipeline

**Workflow:** `.github/workflows/scrape.yml`
**Schedule:** Every 6 hours (`0 */6 * * *`) + manual dispatch
**Runner:** `ubuntu-latest`

Steps:
1. Checkout repo
2. `npm install`
3. `npx puppeteer browsers install chrome`
4. `node build-widget-data.js` — runs all scrapers, outputs `pet-data.json` and updates `adopt-widget.html`
5. Copies `pet-data.json` to `docs/`, writes timestamp to `docs/update-log.txt`
6. Git commit + force push to `main`

**Bot identity:** "WPR Pet Bot" / bot@wausaupilotandreview.com

**GitHub Pages:** Serves from `main` branch `/docs` directory.

## Common Maintenance Tasks

### A scraper returns 0 pets
1. Check `docs/diag-{shelter}-*.html` diagnostic files — open in a browser to see what the page actually returned
2. Common causes: anti-bot block (Cloudflare challenge page), site layout change (selectors no longer match), pagination change
3. The Adoptapet scraper has a fallback path for low results — check if it fired in the Actions log

### Adding a new shelter
1. Write a scraper function in `build-widget-data.js` (use `scrapeAdoptapet()` for aggregator sites, `scrapeNlpac()` for direct HTML sites, or `scrapePetfinder()` for Petfinder)
2. Call it from `main()` and assign to `data.shelters.{key}`
3. Add the shelter key to the `shelterOrder` dedup array
4. In `adopt-widget.html`:
   - Add entry to `SHELTERS` config object
   - Add filter button in `#shelterFilter`
   - Add fallback data in `FALLBACK_DATA`
   - Add footer link
5. Test locally: `node build-widget-data.js` then open `adopt-widget.html`

### Updating fallback data
The `FALLBACK_DATA` object in `adopt-widget.html` (around lines 300-368) contains hardcoded pet listings used when live data can't be fetched. The build script also injects live data into the HTML via regex replacement of the `PET_DATA` const. To refresh fallback data manually, run the scraper and copy representative entries.

### Image URLs
- **Adoptapet:** Uses Cloudinary CDN. Template: `https://media.adoptapet.com/image/upload/c_auto,g_auto,w_400,ar_4:3,dpr_2/f_auto,q_auto/{petId}`
- **Petfinder:** Uses CloudFront. Template: `https://dbw3zep4prcju.cloudfront.net/animal/{animalId}/image/{imageId}.jpg`
- **NLPAC:** Direct URLs from `custompages` path on their site

```bash
# Check diagnostic output if a shelter returned 0

```

### Key Debugging Tips
- The scraper saves diagnostic HTML snapshots to `docs/diag-{shelter}-{context}.html` whenever results are unexpectedly low. Open these in a browser to see exactly what Puppeteer received.
- Adoptapet is a Next.js React app — after JS hydration, pet cards may be re-rendered differently than the server HTML. The fallback path in `scrapeAdoptapet` handles this by extracting URLs from raw page source.
- Petfinder and NLPAC may return bot-detection pages. The `makePage()` stealth settings help but aren't foolproof. If a scraper consistently returns 0, the site may have tightened anti-bot measures.
- The `PET_DATA` injection in `adopt-widget.html` uses a regex replacement (`const PET_DATA={...};`). If the HTML structure around that line changes, the injection may silently fail — the widget will still work but will use stale fallback data instead of fresh scraped data.
